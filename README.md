# Микросервисная система обработки заказов (для тренинга по мониторингу)

Данный проект представляет собой учебный стенд из четырёх микросервисов, инфраструктурных компонентов и экспорта метрик в формате Prometheus. Подходит для отработки навыков настройки мониторинга (Prometheus, Grafana, VictoriaMetrics и др.).

---

## Архитектура

![Архитектура](https://via.placeholder.com/800x400?text=API+Gateway+→+Order+Service+→+RabbitMQ+→+Notification+%26+Analytics)

| Компонент | Технология | Порт | Роль |
|-----------|------------|------|------|
| **API Gateway** | FastAPI (Python) | 8000 | Входная точка: принимает REST-запросы, валидирует, проксирует в Order Service |
| **Order Service** | Flask (Python) | 5001 | Бизнес-логика: создаёт заказы, сохраняет в PostgreSQL, публикует события в RabbitMQ |
| **Notification Service** | Node.js (Express) | 3000 | Потребитель очереди: имитирует отправку уведомлений (email) |
| **Analytics Service** | Python + Prometheus | 8001 | Потребитель очереди: агрегирует статистику (количество заказов, средний чек), выставляет бизнес-метрики |
| **PostgreSQL** | – | 5432 | Хранит заказы |
| **RabbitMQ** | – | 5672, 15672 | Очередь сообщений (обмен событиями) |
| **Redis** | – | 6379 | Кеш для агрегированной аналитики |

## Добавлять новые метрики
* В API Gateway (FastAPI): использовать Instrumentator().instrument(app).expose(app) – легко добавить кастомные метрики через Counter, Histogram и т.д.

* В Order Service (Flask): через PrometheusMetrics можно добавлять свои счётчики.

* В Notification Service (Node.js): библиотека prom-client позволяет создавать любые метрики.

* В Analytics Service: уже есть бизнес-метрики, но можно добавить больше (например, количество заказов по категориям).

## Менять бизнес-логику
* Добавлять новые поля в заказ (например, customer_id).

* Усложнять обработку (валидация, скидки, проверка наличия).

* Расширять уведомления (реальная отправка email через SMTP, Telegram, Slack).

## Подключать дополнительные инструменты
* Grafana – визуализация метрик.

* Prometheus Alertmanager – алертинг.

* Kafka вместо RabbitMQ (или вместе с ним).

* Elastic Stack для логирования.

* Traefik или Nginx в качестве обратного прокси.

## Менять инфраструктуру
* Перевести на Kubernetes (это уже следующий уровень).

* Добавить healthcheck'и и readinessProbe.

* Использовать Helm для деплоя.

## Тестировать нагрузку
* Добавить генератор случайных заказов (например, отдельный сервис или скрипт).

* Прогнать через нагрузочное тестирование (k6, JMeter) и посмотреть, как ведут себя метрики.
